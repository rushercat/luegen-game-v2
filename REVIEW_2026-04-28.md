# Lügen — Comprehensive Review (2026-04-28)

A fresh deep-dive across the whole codebase: live game, beta roguelike, multiplayer rooms, auth/Supabase, and UI/UX. The previous `REVIEW.md` did a lot of work on `beta.js`; this pass picks up where that left off and goes wider — **server.js, client.js, server-beta-rooms.js, beta-mp.js, index.html, auth.js, and the Supabase schema** were all reviewed, and several bugs in those files have not been flagged before.

Every finding has a file/line ref, a severity tag, and a concrete fix. Where I'm uncertain whether a thing is actually broken vs. just suspicious-looking, I've labelled it **"needs verification"** rather than asserting.

> **A correction up front, since I want this report to be trustworthy:** during the audit one of my passes claimed the Gambler character's "forced Cursed card on new floor" mechanic wasn't wired. After cross-checking against the actual code, that claim was **wrong** — `beta.js:1064-1095` and `1184-1185` do wire it. I removed it from the report. Other claims below have been spot-checked against the source.

---

## TL;DR — top items to fix

If you only do five things, do these:

1. **`server.js:828` — null-pointer crash on `lastPlayer.name`** if the play-er disconnects between play and challenge. One client disconnect can brick a whole room. *(critical)*
2. **`server.js callLiar` has no in-flight lock.** Two near-simultaneous LIAR clicks both pass validation and double-mutate room state. *(critical, race)*
3. **`server.js callLiar` validates the challenger but never validates the *winner/loser* are still connected** before dealing the next round. *(high)*
4. **Server-side card validation is hand-only**, not deck-state. A modified client can reuse a card ID after it left the player's hand and the server won't catch it. Combine with the next item below for an actual cheating vector. *(high)*
5. **`auth.js:169-170` — login error messages leak user existence and auth method** (`"Wrong username or password"` vs. `"This account uses Google sign-in, not a password"`). Trivial enumeration oracle. *(high)*

After those: the auth layer needs **rate limiting** and **the OAuth-account password column should be nullable**; the **UI is brutal on screen-readers and color-blind users** (affixes are color-only); and the **biggest design-doc divergences** are the missing **Whisper** character and the wrong starting jokers for **Hoarder** and **Banker**. Detail and ideas below.

## 1. Live game (`server.js` + `public/client.js`)

### 1.1 Critical bugs

#### 1.1.1 `lastPlayer.name` crashes if the playing player disconnects [critical]

`server.js:807-829`

```js
socket.on('callLiar', () => {
  const room = rooms[currentRoomId];
  ...
  const lastPlayer = room.players.find(p => p.id === room.lastPlayerId);
  ...
  io.to(room.id).emit('reveal', {
    cards: lastCards, claimed: room.targetRank, wasLie,
    challengerName: challenger.name,
    lastPlayerName: lastPlayer.name   // ← throws if lastPlayer is null
  });
```

If the player who made the last play disconnects between `playCards` and the challenge, `lastPlayerId` still references their old socket but `find(...)` returns `undefined`. The handler then crashes on `.name`, the room state never advances, and every other player is stuck.

**Fix.** Guard immediately after the `find`:

```js
if (!lastPlayer) {
  return emitError('The last player disconnected — challenge aborted.');
}
```

Then on disconnect, either keep the seat in `players` with `connected: false` (current behavior) but also handle the "player gone" branch in `callLiar`, OR auto-resolve the challenge against the disconnected seat.

#### 1.1.2 `callLiar` has no in-flight lock — simultaneous calls double-resolve [critical]

`server.js:807-884`. There is no `room.challengeInFlight` flag. Two clients calling `callLiar` within the same tick both:

1. pass `room.canChallengeId !== challenger.id`,
2. pass `room.lastPlayedCards.length === 0`,
3. emit `reveal`, run the win/lose logic, mutate hands, and broadcast.

Result: the pile is double-distributed, `currentTurnIdx` advances twice, and clients see a flickering inconsistent state. (Verified: I grepped the file for `challengeInFlight`/`liarInFlight` and there are zero matches.)

**Fix.** First line of the handler:

```js
if (room.challengeInFlight) return;
room.challengeInFlight = true;
try {
  // ... existing logic ...
} finally {
  room.challengeInFlight = false;
}
```

Same pattern needed in any other handler that mutates pile/turn state synchronously after async-ish reads (e.g. `setTargetAndPlay`).

#### 1.1.3 Winner/loser may be a disconnected player [high]

`server.js:832-849`. After the reveal, `winner = wasLie ? challenger : lastPlayer` is used to seed the next round (`startLiarsBarRound`, then `findIndex(p => p.id === starter.id)`). If the winner disconnected between play and challenge, the next round deals to and waits on a ghost seat.

**Fix.** Before starting the next round, fall through to the next *connected and alive* player:

```js
let starter = winner.connected !== false && winner.alive !== false
  ? winner
  : room.players.find(p => p.alive !== false && p.connected);
```

#### 1.1.4 `findNextActiveIdx` clears `isSkipped` mid-loop with side effects [high]

`server.js:415-432`. The function pushes a log line *and* clears `isSkipped` *inside* the loop while still iterating. If two callers invoke `findNextActiveIdx` in the same tick (for example, a disconnect handler races a `callLiar`), the second call sees a different seat as no-longer-skipped, and you get unintended turn advancement.

**Fix.** Two-pass: scan to pick the next idx, then mutate `isSkipped` exactly once. Or only clear `isSkipped` from the actual control-flow site that consumes it.

### 1.2 Validation / security

#### 1.2.1 Card-ID validation trusts the client's hand snapshot [high]

`server.js:775-780`. Validation is "is this id currently in `player.hand`?". That's correct *for the current hand*, but card IDs derive from a deterministic format (rank+suit, plus `JK<idx>`), so a malicious client knows the IDs of cards in the *pile* and the *deck*. There's no per-deal fresh ID, and there's no global "where does this card live" tracker.

Today the bug is mostly latent because the find-in-hand check catches it — but if any future code path holds a stale hand reference, a forged ID could slip through.

**Fix.** Two complementary changes:

- Generate a per-shuffle nonce: `card.id = ${rank}${suit}_${shuffleId}_${idx}`. After a shuffle, all old IDs are dead.
- Maintain a `Map<cardId, location>` (`'hand:p3'`, `'pile'`, `'burn'`, `'deck'`) and validate in one place.

#### 1.2.2 Shuffle uses `Math.random()` [medium / security]

`server.js:225-230`, `271-276`. Both `shuffle` and `makeRoomId` use `Math.random()`. For a casual card game this is fine, but if you ever add **ranked** play or competitive PvP it becomes a real concern: V8's `Math.random` is xorshift128+ and is reverse-engineerable from a few outputs. A determined attacker could in principle predict future deals.

**Fix.** `crypto.randomInt(...)` for shuffles and room codes:

```js
const { randomInt } = require('crypto');
function shuffle(arr) {
  for (let i = arr.length - 1; i > 0; i--) {
    const j = randomInt(0, i + 1);
    [arr[i], arr[j]] = [arr[j], arr[i]];
  }
  return arr;
}
```

#### 1.2.3 Room code entropy is ~23 bits [medium]

`server.js:271-276`. 5 chars from a 30-char alphabet ≈ 24 million combinations. Trivial to brute-force lobby codes in seconds with concurrent socket connections. For private rooms / friend lobbies this is the difference between "you can join my game" and "anyone with a script can crash my game."

**Fix.** Bump to 6 chars (715M combos, ~29 bits) or 7 chars (~34 bits). Add a per-IP rate limit on `joinRoom` regardless.

#### 1.2.4 No rate limiting on socket events [medium / DoS]

There is no rate limit on `playCards`, `callLiar`, or `chat`. A malicious client can flood any of these and exhaust the event loop. `chat` truncates to 200 chars but doesn't throttle, so spam-chat is uncapped.

**Fix.** Token-bucket per-socket: e.g. `chat` 5/5s, `callLiar` 3/5s, `playCards` 5/10s. Disconnect on sustained abuse.

#### 1.2.5 Empty-room cleanup timer can outlive a room [low]

`server.js:494-505`. `setTimeout` references the room by `room.id`. If by some unlucky path `makeRoomId` produces the same code (unlikely but non-zero) before the timer fires, the new room could be deleted on top of itself.

**Fix.** Capture `room.generation = Date.now() + Math.random()` and check it inside the timer.

### 1.3 Gameplay correctness

#### 1.3.1 Jack curse uses `=== 4` instead of `>= 4` [low]

`server.js:438`. The check is exactly four Jacks. If any future affix or consumable can land a fifth Jack into a hand without a turn boundary (e.g. a future "force a Jack into target's hand" item), the curse misses.

**Fix.** `>= 4` is the safe form and matches the design's "holding 4 Jacks at end of any turn = eliminated."

#### 1.3.2 `pullTrigger` doesn't bottom-clamp `chambers` [low]

`server.js:886-898`. Defensive only. If chambers ever goes negative through some other path, the math degenerates.

**Fix.** `player.chambers = Math.max(0, player.chambers ?? GUN_CHAMBERS);` at the top.

### 1.4 UX / polish

- **Mystery Hands reveals counts at hand-empty.** `server.js:360`. Once a player empties their hand the count goes from `null` to `0`, signalling who won before reveal. Probably fine but noticeable.
- **`Rotates in N` label is ambiguous.** `client.js:789-792`. "Whose plays count?" — recommend `"Rotates in N plays"` and a live counter `2 / 5`.
- **Truncated chat is silent.** `server.js:1015-1017`. 500-char message becomes 200 chars with no notice. Add `maxlength="200"` on the input or echo back a truncation warning.

## 2. Beta roguelike (`beta.js`) and beta multiplayer (`server-beta-rooms.js`, `beta-mp.js`)

The previous review fixed an enormous amount of `beta.js` — personalities, Auditor, Lugen, Cleanse, Reward, Hand Mirror, Black Hole picker, Vengeful Spirit, Forger lock, Steel-Jack double, Scapegoat single, Foggy reveal, private-peek toast, Resume modal, Mirror/Hollow alts. Spot-checks confirm all of those are still in place.

What remains:

### 2.1 Design-doc divergences

These are the highest-impact items because they shape the player's experience.

#### 2.1.1 Whisper character is missing entirely [high]

`beta.js:42-127`. `CHARACTER_CATALOG` has rookie / sharp / hoarder / banker / bait / gambler / magician / engineer / witch — but **no `whisper`**. The design doc (lines 241) defines:

> **The Whisper** — Passive: at the start of each round, choose a direction via a toggle button — scout either the previous player or the next player. See one random card from that player's hand. Starts with **Eavesdropper**.

The `bait` character has a `peekAtRoundStart` flag that's *similar*, but it's random target, not directional, and bait already starts with `spikedTrap`. Whisper is conceptually distinct and is named in the design as a key unlock.

**Fix.** Add `whisper` to `CHARACTER_CATALOG` with a `whisperPeek: 'directional'` flag, wire a small toggle UI in round-start (left/right neighbor), and use `eavesdropper` as the starting joker.

#### 2.1.2 Hoarder starts with `slowHand`, design says `Safety Net` [high]

`beta.js:65` — `startingJoker: 'slowHand'`. Design `lugen_design.md:240`:

> **The Hoarder** — Passive: hand size +1, Jack limit 5. Starts with **Safety Net**.

Hoarder's whole identity is "I can hold more, I'm protected from Jack curse." Starting with Safety Net (Jack limit 5) doubles down on that fantasy and synergizes with the +1 hand size; starting with Slow Hand (longer challenge window) is a totally different fantasy.

**Fix.** `startingJoker: 'safetyNet'`.

#### 2.1.3 Banker starts with `surveyor`, design says `The Taxman` [high]

`beta.js:75` — `startingJoker: 'surveyor'`. Design `lugen_design.md:241`:

> **The Banker** — Passive: starts the run with 150g and a Gilded A in the run deck. Starts with **The Taxman**.

Same problem as Hoarder. Banker's identity is "I print money." Taxman gains 10g whenever an opponent picks up a 5+ pile, which extends the Banker's economy fantasy. Surveyor is fine but it's information, not income.

**Fix.** `startingJoker: 'taxman'`.

#### 2.1.4 Counterfeit price/cooldown still mismatched [medium]

`beta.js:461-462` says 35g, the live shop pricing list (`beta.js:615`) corroborates 35g. Design `lugen_design.md:224` says **50g, once per floor**. Implementation is **35g, once per round**.

This was flagged in the prior `REVIEW.md` (1.24) and is still open. Pick one and write it down — they're not equivalent (cheaper-but-cooldown-tighter shifts the item's role from "save it for the boss" to "you'll always have one").

**Fix.** Either bump to 50g and gate via `runState.floorLockedBoughtThisFloor.counterfeit`, or commit to 35g/round and update the design doc.

#### 2.1.5 Round-deck math (still open from prior review, 1.16) [medium]

Design says 30 + N×8 (= 62 at 4 players). Implementation produces ~38. This is a sizeable gameplay difference because Spiked / Spiked Trap / Tracer all live off a fat draw pile. The smaller pile makes those effects fizzle constantly.

**Fix.** Three options listed previously; my recommendation is "raise per-rank cap to 16" — preserves the design's affix-rich feel without expanding the run-deck size from 12 back down to 8.

#### 2.1.6 Treasure nodes (Act III) and 4-fork rotation not present [medium]

The treasure pool is defined (`Shroud`, `Crooked Cards`, `Black Market`, `Gambler's Mark`) but doesn't appear in the fork generator for floors 7 / 8. `lugen_design.md:33`/`36-46` calls these out as Act III specials.

**Fix.** In `showFork`, when `floor === 7` or `floor === 8`, roll a 25% chance to swap Reward for Treasure. The Treasure node opens a free relic from the treasure pool.

### 2.2 Bugs / edge cases

#### 2.2.1 Lugen out-of-turn Liar logic only fires when Lugen is *not* the natural challenger [low / needs verification]

`beta.js:2030-2046`. The branch is gated by `lugenIdx !== challenger`. If Lugen's seat *is* the natural next-player, the special "out-of-turn" behavior never engages — Lugen only gets a normal challenge, and the once-per-round flag is never consumed. That's defensible (no double-call needed) but it means the design's "out-of-turn challenge once per round" cap is effectively soft when Lugen is sitting in the right seat.

**Fix or doc.** Either deliberate (write a comment explaining "out-of-turn = not the natural challenger") or unify the once-per-round cap so Lugen's challenge window probability scales whether they're natural or not.

#### 2.2.2 Eavesdropper still hardcoded for human-at-seat-0 [low, known]

Prior review noted it; deferred for solo. Worth re-flagging because the same code path runs in PvP and there it *is* wrong — the "previous player" should be relative to the local player, not always seat 0.

**Fix.** Replace `prevIdx = 0` (or wherever) with `prevIdx = (mySeat - 1 + N) % N`.

### 2.3 Multiplayer-specific (server-beta-rooms.js + beta-mp.js)

This is the lightest-audited area in `REVIEW.md`. Most issues are about hardening the server now that the rules are right.

#### 2.3.1 No reconnect / resume in beta-mp.js [high]

There is no reconnect handler. If a player drops mid-round, their seat sits there forever and the round can't resolve. In `beta.js` (solo) you have `resumeBetaRun()`; nothing equivalent on the room socket.

**Fix.** On `connect`, send `beta:resume { roomId, authToken }`. Server checks if a seat with that user's id exists in the room, marks it `connected = true`, re-emits room state to that socket only.

#### 2.3.2 Server-side validation likely incomplete on `beta:play` [high / needs verification]

`beta-mp.js:~1114` emits `socket.emit('beta:play', { cardIds: ... })`. Spot-check: `server-beta-rooms.js` should verify (a) it's actually this player's turn, (b) cardIds are in the player's hand, (c) count is in range, (d) target rank isn't locked. The current shape is likely partial — at minimum, audit that all four checks are there, because the server is the only place trust lives.

**Fix.** Centralize validation in a `validatePlay(room, player, cardIds)` helper used by every play path.

#### 2.3.3 Same `callLiar` race exists in beta-rooms [high]

Same issue as live (1.1.2). The bot challenge timeout firing in the same tick as a human click means double-resolution.

**Fix.** Same `challengeInFlight` flag pattern.

### 2.4 AI / boss behavior — verified working

I want to give credit because these were nontrivial fixes:

- Auditor cadence locks `[1..5]` per floor and ticks per challenge opportunity. ✓
- Methodical bluffs at 0.65 only when Jacks ≤ limit-3 *and* matching ≥ 2; else 0.10. ✓
- Mimic mirrors human's last `wasBluff`, falls back to 0.40. ✓
- Wildcard re-rolls bluffRate every play. ✓
- Lugen 7-card start, Jack-limit 6, every play randomly affixed, out-of-turn Liar with scaling probability. ✓
- Mirror copies human's `lastHumanPlay.count` and claim. ✓
- Hollow hides hand size from the player. ✓

These read like personalities now.

## 3. Auth + Supabase (`auth.js`, `setup-supabase.sql`)

### 3.1 Critical / high

#### 3.1.1 Login error messages enable user enumeration [high]

`auth.js:169-170`:

```js
if (!user) throw new Error('Wrong username or password.');
if (!user.password_hash) throw new Error('This account uses Google sign-in, not a password.');
```

Three branches → three observable outcomes:

- "Wrong username or password" → username doesn't exist, OR username exists with wrong password.
- "This account uses Google sign-in" → username exists, OAuth-only.
- success → username exists, password is right.

An attacker can map every username on the platform with one request each.

**Fix.** Collapse all failure branches into the same generic error and the same response time. For OAuth-only accounts, return the generic "wrong username or password" — at most surface the OAuth hint *after* a verified email reset flow.

#### 3.1.2 No rate limit on login / signup [high]

There is no `express-rate-limit` middleware on `/api/login` or `/api/signup`. Combined with 3.1.1 this is a free credential-stuffing channel.

**Fix.**

```js
const rateLimit = require('express-rate-limit');
app.use('/api/login', rateLimit({ windowMs: 60_000, max: 5 }));
app.use('/api/signup', rateLimit({ windowMs: 60 * 60_000, max: 10 }));
```

Add per-account lockout after N failed logins (`login_attempts` table or column).

#### 3.1.3 Password column is `not null` but OAuth users have no password [medium / schema bug]

`setup-supabase.sql:12-13`:

```sql
password_hash text not null,
password_salt text not null,
```

OAuth signup paths in `auth.js` skip these fields entirely. Postgres tolerates this only because `INSERT` omits the column and there's no default — but if anyone runs a migration adding a default, or future code does an upsert with empty strings, accounts break.

**Fix.**

```sql
ALTER TABLE users ALTER COLUMN password_hash DROP NOT NULL;
ALTER TABLE users ALTER COLUMN password_salt DROP NOT NULL;
```

And add a CHECK that exactly one of (password_hash IS NOT NULL, oauth_sub IS NOT NULL) is true:

```sql
ALTER TABLE users ADD CONSTRAINT auth_method_present
  CHECK ((password_hash IS NOT NULL) OR (oauth_sub IS NOT NULL));
```

#### 3.1.4 No unique index on (oauth_provider, oauth_sub) [high]

`setup-supabase.sql:93-94`. Without a unique index, two rows can share the same OAuth identity, and `findOrCreateOAuthUser` will return whichever the planner picks first — sometimes silently linking attackers to other people's accounts.

**Fix.**

```sql
CREATE UNIQUE INDEX users_oauth_unique
  ON users (oauth_provider, oauth_sub)
  WHERE oauth_provider IS NOT NULL;
```

#### 3.1.5 No RLS policies on Supabase tables [medium → critical if anon key ever escapes]

The schema has `users`, `sessions`, etc. with no RLS. Today only the service-role key is used server-side, so it's fine. But "we don't expose the anon key client-side" is a one-typo-away-from-disaster invariant. A defense-in-depth posture turns RLS on now.

**Fix.** Even if all access goes through the server, enable RLS and write owner-only policies. If the anon key is ever shipped to a browser bundle, this becomes the only line of defense between an attacker and every user's session token.

### 3.2 Improvements

- **Token rotation.** Right now a session token is good for 30 days with no rotation. Implement a refresh-on-use pattern with a token family (rotate on every touch, detect concurrent use as a compromise signal).
- **Admin endpoint audit log.** `/api/beta/admin/unlock-all` (`server.js:116-122`) takes effect with no record. Add an `audit_log` table and write `(user_id, ip, action, ts)` for every admin call.
- **Don't expose `is_admin` in `publicUser`** (`auth.js:128`). Information leak — there's no need for clients to know who's an admin.
- **Email verification flow.** Right now any email is taken at face value. Add a verify token + 24h expiry.
- **Password reset.** No reset endpoint exists. For password accounts this is table stakes.
- **Unicode-aware password length** (`auth.js:143`). `password.length` counts UTF-16 units; emoji passwords slip past the 6-char minimum. Use `[...password].length`.
- **OAuth provider whitelist** (`server.js:157`). `provider = supaUser.app_metadata?.provider || 'oauth'` — the fallback `'oauth'` is dangerous because every misconfigured provider collides on the same key. Whitelist explicitly, reject unknown.

### 3.3 Schema nits

- Add indexes: `CREATE INDEX ON users (email) WHERE email IS NOT NULL;` and `CREATE INDEX ON users (oauth_sub) WHERE oauth_sub IS NOT NULL;`.
- Add a CHECK on `beta_run_history`: `CHECK (jsonb_array_length(beta_run_history) <= 50)` so a buggy client can't fill a column with megabytes.

## 4. UI / UX (`public/index.html`)

The visual language is solid (dark bar/casino vibe, the hand layout is readable on desktop, the challenge bar is dramatic). The accessibility and mobile gaps are real and worth a polish pass.

### 4.1 Critical

#### 4.1.1 Affixes are color-only [critical]

Affix rings (Glass = blue, Gilded = gold, Spiked = red, Cursed = purple, Steel = silver, Mirage = rainbow, Hollow = black, Echo = cyan) are differentiated only by color. 8% of male players cannot reliably distinguish red/green/orange. **Glass + Gilded** can look identical to a deuteranope.

**Fix.** Add a 2-letter code on every affixed card (`GLD`, `GLS`, `STL`, `MRG`, `SPK`, `CRS`, `HLW`, `ECH`), positioned in a corner so it's visible on face-down backs too. Optionally a pattern overlay (hatching/dots).

#### 4.1.2 No keyboard escape from modals, no focus trap [critical]

The big z-50 overlays (auth, settings, character pick, shop, fork, reward) don't listen for Escape, don't trap focus, and don't return focus to the trigger button on close. Keyboard-only users can get stuck.

**Fix.** A small `trapFocus(modalEl)` helper:

```js
function trapFocus(el, onClose) {
  const focusables = el.querySelectorAll('button, [href], input, select');
  const first = focusables[0], last = focusables[focusables.length - 1];
  el.addEventListener('keydown', e => {
    if (e.key === 'Escape') return onClose();
    if (e.key !== 'Tab') return;
    if (e.shiftKey && document.activeElement === first) { last.focus(); e.preventDefault(); }
    else if (!e.shiftKey && document.activeElement === last) { first.focus(); e.preventDefault(); }
  });
  first?.focus();
}
```

#### 4.1.3 Challenge timer is visual-only [critical]

The red countdown bar has no `role="progressbar"` and no `aria-live` announcement when the challenge window opens or closes. Screen-reader users will simply miss the call.

**Fix.**

```html
<div id="betaChallengeBar"
     role="progressbar"
     aria-valuemin="0" aria-valuemax="5" aria-valuenow="5"
     aria-label="Time to call LIAR">
```

Plus a hidden `<div role="status" aria-live="assertive" id="gameAnnouncer"></div>` that gets `"5 seconds to call LIAR"` and `"LIAR window closed"` written into it.

### 4.2 Mobile (≤ 380px viewports)

- Hand wraps poorly because cards are 64px and there are five of them. Add `overflow-x-auto` on `#betaHand` for narrow screens, scale cards to `w-12` on `<sm`.
- Tap targets on action buttons (`betaPlayBtn`, `betaLiarBtn`, `betaPassBtn`, etc.) are ~32px tall — under Apple/WCAG 44px guidance. Use `min-h-[44px]` everywhere.
- The settings menu (`w-60` absolute right-0) overflows on phones. Add a `max-w-[calc(100vw-2rem)]` cap and `right-2` padding.
- The 40-line round log eats half the viewport. Use `max-h-24 sm:max-h-40`.

### 4.3 Accessibility

- Icon-only buttons (🏆, ⚙️, 📊, ⚠️, ❤️ LIAR!) need `aria-label`. Emoji are read as their Unicode names by default — "trophy" is okay-ish but "warning sign" is not "report a bug."
- All `<input>` elements use placeholder-only labels. Add real `<label for="...">`.
- Every interactive element with `focus:outline-none` needs a `:focus-visible` ring. Right now keyboard users have no visual feedback at all on hover-styled buttons.
- A polite live region for game events (`role="status" aria-live="polite"`) so screen readers hear "Liar called!", "You won the round", "Glass triggered — 3 cards burned".

### 4.4 Animation / feedback

- **Toast system is missing.** Gold gained, joker triggered, relic acquired, floor complete — these are silent right now (a log line in a 40-line scroll buffer is not "feedback"). A bottom-left toast stack with auto-fade would massively improve game feel.
- **Reveal has no animation.** When LIAR is called, the cards just appear face-up. A flip animation (CSS `transform: rotateY(180deg)` over 400ms) would make the moment land.
- **Audio is unused.** `gunshot.mp3` and `click.mp3` exist but I see no `new Audio()` calls in `index.html`. Plug them into LIAR resolution + every card click. Add a mute toggle in the settings menu.
- **Card selection feedback is subtle.** `translateY(-14px)` is fine but the affix ring doesn't always move with the card on slow GPUs. Force `will-change: transform`.
- **Async buttons don't disable.** Shop purchases and "Create Room" should grey out for the duration of the request to prevent double-submit.

### 4.5 Onboarding

- **No first-run tutorial.** The "How to play" `<details>` element on the lobby is the only explanation; new players miss it. A lightweight 5-step overlay (`localStorage.setItem('lugenSeenTutorial', '1')`) walks through: deal → Target Rank → play face-down + claim → 5s LIAR window → Jack curse.
- **Affix legend is buried.** Affix catalog is in the beta menu, not in-game. Add a `?` button next to each affix ring that pops a one-line definition.
- **Character cards lack stat displays.** Names and avatars are visible but the passive isn't shown until hover. Show passive + starting joker on the card itself.

### 4.6 Polish

- Standardize button copy: "Sign In" vs "Sign in", "Launch Beta Testing" vs "Start Round", "Create Room" vs "Make Room".
- `title="..."` attributes on tooltips are invisible on touch. Mirror to `aria-label` and surface on tap.
- Add a "current player" glow on the opponent grid — right now you have to read the turn label to know whose turn it is.

## 5. Gameplay & balance ideas

These are subjective — pick what excites you. I tried to keep each one cheap to implement.

### 5.1 Things I'd rebalance now

- **Counterfeit:** commit to 50g / once-per-floor (matches design) so the player has to *time* it rather than mash it every round. Right now it's used reflexively the moment Target rotates to a rank you don't have.
- **Round-deck math:** raise per-rank cap to 16. Smallest fix that brings the deck near design size, makes Spiked/Tracer reliable, and reduces fizzle frustration.
- **Sleight of Hand:** 2g cost to draw is quietly *strong* in a 5-card-hand game. Consider once-per-round AND only on a turn you actually play (not on a pass) so it can't be used to stall indefinitely.
- **Vengeful Spirit:** the rework (preload 2 forced Jacks into the targeted bot's hand next round, cap at limit-1) is good but in 2-player Act III it's basically "win the next round." Cap at 1 forced Jack at 2-player tables.
- **Greedy modifier (Jack limit 3):** at 3-cap, the Jack-fairness rule caps starting Jacks at 2, meaning you start with at most 2/3 of your Jack budget. Most rounds that's a soft pre-curse. Consider giving the Greedy modifier one free "pop the curse" pulse per round (delete one Jack from your hand for free at start).

### 5.2 New jokers

The previous review's joker list is good. Three more I'd add:

| Name | Rarity | Effect |
|---|---|---|
| **Caller's Mark** | Uncommon | The first time each round you call LIAR and are right, gain 15g. Encourages skilled calling, not spam. |
| **Stage Light** | Rare | When the Target rotates, you privately see the new Target 1 second before everyone else. (Counterplay to Counterfeit; rewards positional play.) |
| **The Rookery** | Legendary | Once per floor: name a rank. For the rest of the round, every card of that rank in any hand is publicly revealed. Loud, asymmetric, fun. |

### 5.5 New characters

- **The Mute** (already in the prior review) — can't call LIAR but immune to Cursed. +1 joker slot. Identity: pure offense, no defense.
- **The Twin** — start the run with two copies of one chosen rank (4 cards) in your run deck. Synergy: Forger / Distillation builds.
- **The Apostate** — every run-deck card you own loses its affix at the start of the round and gains a *random* affix. Roll the dice every round. Identity: variance-loving, Wildcard-coded.

### 5.6 New modifiers

- **Inverted** — Jacks ARE the Target this round. Other ranks are bluffs. (Already in prior review's wishlist; I love this and would prioritize it. The Jack curse becomes an asset.)
- **Open Books** — One round per floor: all hands public. Pure tactics, no bluff. Works great in Act III where reads are gone.
- **Last Call** — Each player has 5 turns total this round. Whoever has the most cards left after 5 turns each *picks up the pile*. Forces aggression.

### 5.7 New events

- **Mysterious Stranger** — trade one run-deck card for a random affixed card from a hidden pool.
- **Card Sharp** — pay 50g to peek at next floor's modifier *and* opponents' personalities.
- **Shrine of Hearts** — 100g for a Heart shard.
- **Drunken Brawl** — lose 30g, gain a free Counterfeit.
- **The Auditor's Apprentice** — pay 80g to lock in your fork choice for next floor (commit before seeing options).

### 5.8 Achievements

The prior review's achievement list is great — I'd add three with high "I want to chase this" energy:

- **The Mind-Reader** — call LIAR correctly 5 times in a row in a single floor. *Unlocks*: a glowing eye card-back skin.
- **The Ghost** — win a round without playing a single card (everyone else folds via skips/curse). *Unlocks*: invisible-card skin.
- **The Fox** — bluff every play in a single round and survive. *Unlocks*: foxtail card-back.

### 5.9 Quality-of-life

- **Run-seed sharing.** Generate a seed at run start, shuffle deterministically from it. Players can share and compare runs ("seed 4F2K-9A7B"). Massive for community feel.
- **End-of-run summary.** Floors cleared, total gold, jokers used, achievements unlocked, bossses fought. Players love this.
- **Spectate mode.** When you're eliminated in PvP, watch the rest play. No new mechanics, just don't disconnect them.

## 7. What's working really well

A short list because it's worth saying out loud:

- The personality AI is excellent now — Methodical actually feels methodical, Wildcard actually feels chaotic, Lugen actually feels final.
- The relic and fork systems flow into each other cleanly. Players make real choices.
- Run-deck colored borders + ownership tracking are a smart design that nobody else in the bluffing-card space is doing.
- The boss design (Auditor as counting puzzle, Cheater as pattern recognition, Lugen as math floor) is a *teaching* curve, not just a difficulty curve. That's hard to do.
- The codebase is readable. Variable names match the design vocabulary. Future-you will thank present-you.

The bones are stronger than I expected for a game still calling itself v0.1. Pick the top of §6 and ship a v0.2 — most of the friction is in fixable corners, not in the design.
